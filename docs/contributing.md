# Contributing

This document is for contributors to `es-typed-go`. The top-level `README.md` is intentionally focused on library users. Development workflow, repository rules, internal tools, and maintenance notes belong here.

## Scope

Use this guide when you are:

- changing library code
- adding tests
- updating generators
- editing contributor-facing docs
- maintaining v8/v9 parity
- working on repository tooling

## Development prerequisites

- Go `1.26` or later
- Docker and Docker Compose
- A local Elasticsearch environment for integration tests

## Repository layout

```text
estype/           Core shared types
esv8/             Elasticsearch v8 wrapper
  query/          Query/sort/aggregation builders for v8
  generator/      API coverage/spec generator for v8
esv9/             Elasticsearch v9 wrapper
  query/          Query/sort/aggregation builders for v9
  generator/      API coverage/spec generator for v9
cmd/estyped/      CLI for generating typed field constants
docs/             User and contributor documentation
examples/         Example applications
tools/            Internal contributor tools
```

## Build and test commands

### Build all packages

```bash
go build ./...
```

### Run unit tests

```bash
go test ./... -short -count=1
```

### Run unit tests with verbose output

```bash
go test -v -count=1 -timeout=60s ./...
```

### Run integration tests

These require running Elasticsearch instances.

```bash
go test -tags=integration -v ./estype/... ./esv8/...
go test -tags=integration -v ./esv9/...
```

### Regenerate generated files

```bash
go generate ./...
```

## Local Elasticsearch for integration tests

The repository includes `compose.yaml` for local development. Start the services before running integration tests.

```bash
docker compose up -d
docker compose ps
```

The default local ports are:

- Elasticsearch v8: `http://localhost:19200`
- Elasticsearch v9: `http://localhost:19201`

## Linting and custom vet checks

This repository includes a custom analyzer for unchecked type assertions.

Build it:

```bash
go build -o ./bin/okassertcheck ./tools/okassertcheck
```

Run it with `go vet`:

```bash
go vet -vettool=$(pwd)/bin/okassertcheck ./...
```

A practical local validation sequence is:

```bash
go test ./... -short -count=1
go build ./...
go build -o ./bin/okassertcheck ./tools/okassertcheck
go vet -vettool=$(pwd)/bin/okassertcheck ./...
```

## Generator workflows

### Regenerating API coverage and spec files

The v8 and v9 packages include generator code used to maintain full API coverage artifacts.

Run:

```bash
go generate ./...
```

If you change generator logic, verify both generated output and affected tests.

### Generating field constants from a mapping

```bash
go tool estyped -mapping mapping.json -out esmodel/fields.go -package esmodel
```

### Generating field constants from a Go struct

```bash
go tool estyped -struct Product -out product_fields.go
```

When using struct mode, the generator reads `Mapping()` to infer Elasticsearch field type names for comments.

## Coding conventions

### Type safety

Always use the repository’s typed wrappers where appropriate:

- `estype.Field`
- `estype.Index`
- `estype.Alias`

Do not replace them with bare `string` in public APIs unless there is a strong reason.

### Error wrapping

Use:

```go
fmt.Errorf("context: %w", err)
```

### Logging

Use `log/slog` only.

### Tests

- Use `gotest.tools/v3/assert`
- Put `t.Parallel()` at the top of every test and sub-test
- For integration tests, use the `integration` build tag
- In tests, use generic field names only:
  - `status`
  - `title`
  - `category`
  - `tags`
  - `items`
  - `date`
  - `price`
  - `id`
  - `name`
  - `type`
  - `enabled`
  - `value`

### Generated files

Generated files must include:

```go
// Code generated by estyped; DO NOT EDIT.
```

## v8 / v9 parity rules

Changes under `esv8/` usually need the same change under `esv9/`.

When you add or modify:

- query helpers
- sort helpers
- aggregation builders
- property builders
- typed search behavior
- create-index helpers
- client interface behavior

check whether the equivalent v9 file must be updated as well.

Do not let the two versions drift accidentally.

## Search API guidance

For application-facing examples and documentation:

- prefer `Search[T](...)` for normal typed searches
- use `SearchDocuments(...)` when only decoded document sources are needed
- for single-result retrieval, use `Search[T](...)` or `SearchDocuments[T](...)` with `Limit(1)` / `Size: 1`
- reserve `SearchRaw(...)` for advanced escape-hatch scenarios

When using `query.NewSearch().Build()`, you can pass the resulting `query.SearchParams` directly to `Search[T](...)` and `SearchDocuments[T](...)`.

## Property builder guidance

All property constructors should follow the functional-options pattern already used in the repository.

Example shape:

```go
prop := esv8.NewTextProperty(
    func(p *types.TextProperty) { p.Analyzer = ptr("kuromoji") },
)
```

If you add a new property type wrapper:

1. update `esv8/property.go`
2. update `esv9/property.go`
3. add tests in the appropriate package

## Adding new query helpers

When adding a new query helper:

1. update `esv8/query/helpers.go`
2. update `esv9/query/helpers.go`
3. add matching tests in both version packages when appropriate

## Documentation rules

- `README*` files are for library users
- contributor and maintenance details belong under `docs/`
- avoid emoji
- avoid excessive bold for inline emphasis

If a change only matters to contributors, do not put it in the main README.

## Session notes

After each meaningful work step, add or update a compressed session note under `.sessions/`.

Recommended filename shape:

```text
.sessions/yyyymmddhhmmss_title.md
```

The goal is to make the current context resumable.

## Recommended contributor workflow

1. make the code change
2. update the mirrored v8/v9 implementation if needed
3. update tests
4. regenerate code if required
5. run unit tests
6. run integration tests when the change affects runtime behavior
7. run full build
8. update docs under `docs/` if contributor guidance changed
9. add or update a session note

## Pull request checklist

Before opening a PR, verify:

- [ ] user-facing docs remain user-focused
- [ ] contributor-only details are in `docs/`
- [ ] v8 and v9 remain aligned where expected
- [ ] generated files are regenerated if needed
- [ ] unit tests pass
- [ ] integration tests pass when relevant
- [ ] `go build ./...` passes
- [ ] custom vet/analyzer checks pass where relevant
- [ ] session notes were updated
