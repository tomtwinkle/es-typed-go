## 核心设计原则

### 类型安全优先

`estype.Field`、`estype.Index`、`estype.Alias` 是不同的字符串类型。  
贡献代码时，请保持这种类型边界，不要在公开 API 中退回为裸 `string`，除非确实是在表达原始 JSON key 或类似概念。

### v8 / v9 对称性

`esv8/` 与 `esv9/` 应尽量保持一致。

如果您修改了以下任一内容，通常也应同步修改另一个版本：

- 查询辅助函数
- 排序构建器
- 聚合构建器
- property 构造器
- typed search 相关 API
- create index / alias / document 等包装逻辑
- 测试

如果某个变更只能作用于其中一个版本，请在提交说明或注释中明确说明原因。

### 面向应用代码的 API 与逃生口分离

- 优先维护高层、易用、类型安全的 API
- 保留 `SearchRaw(...)` 这类低层逃生口，以支持暂未建模的 Elasticsearch 请求形状
- 不要为了覆盖边缘场景而让主要 API 退化为“全是 `any` 的薄封装”

## 代码生成

### `estyped` CLI

`cmd/estyped/` 是字段常量生成器。

它支持两种主要输入：

- Elasticsearch mapping JSON
- 带 JSON tag 的 Go struct

常见用法：

```/dev/null/es-typed-go/docs/contributing.zh-CN.md#L1-999
go tool estyped -mapping mapping.json -out model.go -package model
go tool estyped -mapping mapping.json -out model.go -package model -group Sample
go tool estyped -struct MyType -out model.go -package model
```

### 生成文件约定

所有生成文件都应包含头部：

```/dev/null/es-typed-go/docs/contributing.zh-CN.md#L1-999
// Code generated by estyped; DO NOT EDIT.
```

不要手动编辑生成文件后再提交为“源码变更”；如果需要改动生成结果，应优先修改生成逻辑并重新生成。

### 结构体模式与 `Mapping()`

当 `estyped -struct` 需要推断字段的 Elasticsearch 类型名时，约定由结构体实现：

```/dev/null/es-typed-go/docs/contributing.zh-CN.md#L1-999
Mapping() estype.Mapping
```

如果缺少该方法，生成注释中的字段类型可能退回为 `"unknown"`。  
贡献者在调整 struct mode 行为时，应同时考虑：

- CLI 行为
- README 中面向使用者的说明
- 相关测试

## 代码风格与约定

### 错误处理

统一使用错误包装：

```/dev/null/es-typed-go/docs/contributing.zh-CN.md#L1-999
fmt.Errorf("context: %w", err)
```

避免丢失上下文的裸返回，除非调用点已经足够明确。

### 日志

统一使用 `log/slog`。  
不要在新代码中引入其他日志框架，除非有非常明确且仓库级别认可的理由。

### 测试断言

测试中统一使用：

- `gotest.tools/v3/assert`

例如：

```/dev/null/es-typed-go/docs/contributing.zh-CN.md#L1-999
assert.NilError(t, err)
assert.Equal(t, got, want)
```

### 并行测试

每个测试函数与子测试开头都应调用：

```/dev/null/es-typed-go/docs/contributing.zh-CN.md#L1-999
t.Parallel()
```

如果某个测试不能并行运行，请在代码中明确说明原因。

### 测试字段命名

测试里请尽量使用通用字段名，避免引入与业务无关的噪音。优先使用：

- `status`
- `title`
- `category`
- `tags`
- `items`
- `date`
- `price`
- `id`
- `name`
- `type`
- `enabled`
- `value`

### 文档风格

- 不使用 emoji
- 不要过度使用粗体强调
- README 以使用者视角为主
- 贡献、维护、生成、工具、内部规则放在 `docs/` 下的贡献文档中

## 常见修改场景

### 新增 property 类型包装器

1. 在 `esv8/property.go` 中添加 `NewXxxProperty(...)`
2. 在 `esv9/property.go` 中做相同修改
3. 遵循现有 functional-options 模式
4. 增加对应测试

### 新增查询辅助函数

1. 修改 `esv8/query/helpers.go`
2. 同步修改 `esv9/query/helpers.go`
3. 在对应测试文件中补充测试
4. 保持 v8 / v9 API 一致

### 修改 typed search

修改 `typed_search.go` 或相关 builder / response 结构时，应重点关注：

- 高层 API 是否仍然易用
- 是否仍保持类型安全
- 是否破坏 v8 / v9 对称性
- 是否需要更新 README 中的示例
- 是否需要补充聚合访问器相关测试

### 修改 ESClient / ESClientSpec

如果您修改：

- `ESClient`
- `ESClientSpec`
- create index / alias / document / reindex 等接口或实现

请同步检查：

- 两个版本目录
- 集成测试
- 相关文档示例
- 生成代码或覆盖测试是否需要更新

```/dev/null/es-typed-go/docs/contributing.zh-CN.md#L1-999

```

如果改动涉及运行时行为、Elasticsearch 请求/响应、索引管理、别名、reindex 或 typed search，建议再执行：


```/dev/null/es-typed-go/docs/contributing.zh-CN.md#L1-999

```
