# コントリビューションガイド

このドキュメントは `es-typed-go` の開発・保守に参加するコントリビューター向けのガイドです。  
ライブラリの使い方については `README.ja.md` を参照してください。

## 目的

`es-typed-go` は、Elasticsearch 向けの型安全な Go ラッパーとして、以下を重視しています。

- コンパイル時に誤用を防ぐ API 設計
- `esv8/` と `esv9/` の高い機能対称性
- 利用者向け API の分かりやすさ
- 型安全なフィールド・クエリ・マッピング定義
- 自動生成コードと手書きコードの責務分離

コントリビュート時は、新機能の追加だけでなく、既存の設計方針との整合性も重視してください。

## リポジトリ構成

主要なディレクトリは以下のとおりです。

```text
estype/           コア共有型
esv8/             Elasticsearch v8 ラッパー
esv9/             Elasticsearch v9 ラッパー
cmd/estyped/      フィールド定数生成 CLI
docs/             利用者向け / 開発者向けドキュメント
examples/         サンプルコード
tools/            開発補助ツール
```

補足:

- `estype/` には `Field`、`Index`、`Alias`、`Mapping`、`Settings` などの共有概念があります
- `esv8/` と `esv9/` は、可能な限り同じ API 形状を維持します
- `cmd/estyped/` は mapping JSON や Go struct から型付き Field 定数を生成します
- `tools/` にはリポジトリ内部の検証や保守のための補助ツールがあります

## 開発環境

前提:

- Go 1.26 以上
- Docker / Docker Compose
- Elasticsearch v8 / v9 の動作確認環境

ローカルでの基本確認コマンド:

```bash
go build ./...
go test ./... -short -count=1
go test -v -count=1 -timeout=60s ./...
```

integration test を実行する場合:

```bash
go test -tags=integration -v ./estype/... ./esv8/...
go test -tags=integration -v ./esv9/...
```

## 開発の基本方針

### 1. 型安全性を優先する

`estype.Field`、`estype.Index`、`estype.Alias` は、単なる文字列ではなく、誤用防止のための distinct type です。  
新しい API を追加するときは、可能な限り裸の `string` ではなく既存の専用型を使ってください。

例:

- フィールド名は `estype.Field`
- インデックス名は `estype.Index`
- エイリアス名は `estype.Alias`

### 2. 利用者向け API では分かりやすさを重視する

このライブラリは、Elasticsearch 公式 typed client の「型はあるが使いにくい」という課題を改善することを目的にしています。  
そのため、公開 API では次を重視してください。

- シンプルで読みやすい関数シグネチャ
- コンパイル時に誤用が分かる設計
- `any` の露出を最小限にすること
- 明確な命名

### 3. `esv8/` と `esv9/` の parity を保つ

`esv8/` に対する変更は、多くの場合 `esv9/` にも同じ変更が必要です。  
どちらか一方だけに機能や修正が入っていないかを必ず確認してください。

特に次の変更では parity を意識してください。

- query helper の追加・修正
- property builder の追加・修正
- typed search API の変更
- create index 系 API の変更
- テストの追加

### 4. escape hatch と高レベル API を分ける

通常の利用者には高レベル API を提供し、複雑な Elasticsearch リクエストが必要な場合のみ低レベル API を使う方針です。

- 通常は `Search[T](...)`
- ドキュメントだけ必要なら `SearchDocuments[T](...)`
- 1 件だけ欲しい場合も `Search[T](...)` または `SearchDocuments[T](...)` に `Limit(1)` / `Size: 1` を指定する
- 件数だけ必要なら `Count(...)` を優先する
- 特殊な形のリクエストが必要なら `SearchRaw(...)`

`query.NewSearch().Build()` は `query.SearchParams` を返すため、そのまま `Search[T](...)` や `SearchDocuments[T](...)` に渡せます。

新しい API を追加するときも、この役割分担を崩さないようにしてください。

## コーディング規約

### 命名と型

- フィールド名は常に `estype.Field`
- エラーは `fmt.Errorf("context: %w", err)` でラップ
- ログは `log/slog` を使用
- 不要に `string` へ戻さない
- 公開 API 名は利用者視点で簡潔にする

### テスト

- アサーションは `gotest.tools/v3/assert` を使用
- 各テスト関数の冒頭で `t.Parallel()` を呼ぶ
- サブテストでも `t.Parallel()` を使う
- integration test には `//go:build integration` を付ける
- テストで使うフィールド名は汎用的なものに限定する

推奨するテスト用フィールド名:

- `status`
- `title`
- `category`
- `tags`
- `items`
- `date`
- `price`
- `id`
- `name`
- `type`
- `enabled`
- `value`

### ドキュメント

- README はライブラリ利用者向けに保つ
- 開発手順や内部ツール説明は `docs/` に置く
- 過剰な装飾は避ける
- 絵文字は使わない

## 変更時のチェックポイント

### Query helper を追加する場合

1. `esv8/query/` に helper を追加
2. `esv9/query/` に同等の helper を追加
3. 対応する unit test を両方に追加
4. 公開 API の一貫性を確認

### Property builder を追加する場合

1. `esv8/property.go` に `NewXxxProperty(...)` を追加
2. `esv9/property.go` に同等の実装を追加
3. functional-options パターンに従う
4. 対応テストを追加
5. 必要なら property reference を更新

### `estype/` の共有型を変更する場合

`estype/` の変更は `esv8/`、`esv9/`、`cmd/estyped/`、README、examples まで波及することがあります。  
特に以下を確認してください。

- 公開 interface の変更がないか
- generator 側の期待値が変わらないか
- examples が古くなっていないか
- README と `docs/` に矛盾がないか

### `cmd/estyped/` を変更する場合

ジェネレータ変更は利用者体験に直結するため、以下を必ず確認してください。

- mapping JSON からの生成結果
- struct mode の生成結果
- コメントや出力フォーマットの互換性
- 生成コードの可読性
- テストカバレッジ

生成ファイルには必ず以下のヘッダーを含めます。

```go
// Code generated by estyped; DO NOT EDIT.
```

## Lint / 静的検証

このリポジトリでは、型アサーションの unsafe な利用を検出するための補助チェッカーを利用できます。

```bash
go build -o ./bin/okassertcheck ./tools/okassertcheck
go vet -vettool=$(pwd)/bin/okassertcheck ./...
```

このチェッカーは主に以下を検出します。

- `ok` を確認しない型アサーション
- 圧縮されすぎた危険な型アサーション利用
- テストコードや通常コードで許可していないパターン

ただし、最も重要なのは「通すこと」ではなく、意図が明確で安全なコードを書くことです。

## Integration test について

integration test は実際の Elasticsearch インスタンスを使って、以下のような観点を確認します。

- create index / delete index
- alias lifecycle
- document CRUD
- search
- aggregation
- reindex
- create-index helper
- spec client の基本動作

変更内容が runtime behavior に影響する場合は、関連 integration test を実行してください。

## README と docs の役割分担

今後のドキュメント方針は以下です。

### README
利用者向け:

- ライブラリの目的
- インストール方法
- クイックスタート
- 公開 API の使い方
- 主要ドキュメントへのリンク

### docs
開発者・保守者向け、または詳細リファレンス:

- コントリビューション手順
- 開発ルール
- 内部設計メモ
- property reference
- 将来的な詳細設計資料

利用者に不要な内部情報は README に戻さないようにしてください。

## 変更の進め方

推奨フロー:

1. 変更対象を特定する
2. `esv8/` と `esv9/` の両方への影響を確認する
3. 必要な unit test を追加する
4. 必要なら integration test を実行する
5. README / docs / examples の整合性を確認する
6. 生成コードや generated test への影響があれば更新する

## セッションノート運用

意味のある作業ステップごとに、圧縮したセッションノートを `.sessions/` 配下へ保存します。

ファイル名形式:

```text
.sessions/yyyymmddhhmmss_title.md
```

セッションノートには、少なくとも以下を残してください。

- 何を変更したか
- なぜ変更したか
- 影響範囲
- 実行したテスト / ビルド
- 次に続ける人が必要な文脈

これにより、作業の再開や引き継ぎが容易になります。

## 破壊的変更の扱い

破壊的変更を入れる場合は、次を明確にしてください。

- 利用者向け API にどう影響するか
- README の例が古くならないか
- examples のコードが壊れないか
- generator や integration test が追従しているか
- v8 / v9 の parity が維持されているか

必要であれば、README や docs に移行方法を追記してください。

## Pull Request 前の推奨確認

少なくとも以下を確認することを推奨します。

```bash
go build ./...
go test ./... -short -count=1
```

変更内容に応じて追加で:

```bash
go test -tags=integration -v ./estype/... ./esv8/...
go test -tags=integration -v ./esv9/...
go vet -vettool=$(pwd)/bin/okassertcheck ./...
```

## 質問や提案について

大きな API 変更やドキュメント方針の変更は、いきなり広範囲に編集する前に、目的と設計意図が共有できる形で進めるのが望ましいです。

特に以下は事前に方針を揃える価値があります。

- 公開 API の命名変更
- `estype/` の基盤 interface の変更
- generator の出力仕様変更
- README の情報設計変更
- `esv8/` / `esv9/` parity に関わる設計判断
